# 00. Constitution

> AIを管理するAIではない。知識を管理するOSである。
> AIが変わっても、知識は変わらない。
> 知識を資産に変える。検証を標準化する。成果物を自動生成する。知識を継続的に進化させる。

## N-HUB-OS が存在する理由

海外発の一次情報(Claude Code, MCP, AI Agent, Browser Automation, AI Video Generation, AI Coding, AI SaaS, Immersive Web など)は流量が多く鮮度が命だが、そのままでは「見た」だけで終わり資産にならない。N-HUB-OS は、収集した情報を **検証 → 構造化 → 再利用可能な成果物(skill / MCP / 記事 / 動画 / 自動化ワークフロー)** に変換し続けるための作業場である。

## N-HUB-OS と N-Link の関係

- **N-Link** = 脳(記憶・承認ガバナンス・品質ゲート・タスク人格)。`n-link-brain` MCP として提供される。
- **N-HUB-OS** = その脳を使って実際に手を動かす作業場(情報収集・検証・知識化・成果物生成)。

N-HUB-OS は N-Link の代わりに記憶や承認の仕組みを自前で作らない。判断・記憶・品質保証はすべて N-Link に委譲し、N-HUB-OS 側は「何を収集し、何を作るか」というドメイン知識と成果物に集中する。両者のフォルダ・データストアは明確に分離し、混在させない([CLAUDE.md](../CLAUDE.md) の絶対ルール2を参照)。

## 核となる考え方

1. **貯めるより検証する。** 未検証の情報は `brain/research/` に留め、複数ソースで裏取りできたものだけが `brain/verified/` に昇格する。
2. **一度作った知識・skill は使い回す。** 同じ調査・同じ判断を二度しない(重複排除・再利用を最優先する)。
3. **過剰実装をしない。** 使われない engine やドキュメントのためのドキュメントは作らない。各 Phase は実際にタスクが来たときに初めて実装する(段階的構築、[90_roadmap.md](90_roadmap.md) 参照)。
4. **すべての実装作業は N-Link の pipeline を通す。** `brain_analyze_task` なしに実装を始めない、`brain_approval_check` なしに副作用のある操作をしない、品質ゲートを通さず「完了」と言わない。

## 非目標(Non-Goals)

- N-HUB-OS は N-Link の記憶・承認・監査機能を再実装しない。
- N-HUB-OS は無人で公開・送信・課金を行わない(公開系操作は常に N-Link 経由でオーナー承認を取る)。
- N-HUB-OS は「壮大な構想」を一度に全部実装することを目指さない。実装は要求駆動・段階的に行う。
